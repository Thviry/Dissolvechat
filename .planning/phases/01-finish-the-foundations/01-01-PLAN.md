<objective>
Add zero-byte bucket padding to the E2EE layer so encrypted message payloads are always one of four fixed sizes (512B, 1KB, 2KB, 4KB). This eliminates ciphertext length as a side-channel that leaks message length to a network observer.

Purpose: AES-GCM is a stream mode — ciphertext length equals plaintext length exactly (plus 16-byte auth tag). Without padding, a 20-byte "ok" message and a 1000-byte essay produce visibly different ciphertext sizes. Bucket padding closes this information leak.

Output: Modified `e2ee.js` in both `client/` and `desktop/` with `padPlaintext`/`unpadPlaintext` helpers integrated into `e2eeEncrypt` and `e2eeDecrypt`.
</objective>

```javascript
// Padding helpers to add (above the exported functions):
const BUCKETS = [512, 1024, 2048, 4096];

function padPlaintext(bytes) {
  const len = bytes.length;
  let target = BUCKETS.find(b => b >= len);
  if (!target) {
    target = Math.ceil(len / 1024) * 1024;
  }
  const padded = new Uint8Array(target);
  padded.set(bytes); // remaining bytes are zero-initialized
  return padded;
}

function unpadPlaintext(bytes) {
  let end = bytes.length;
  while (end > 0 && bytes[end - 1] === 0) end--;
  return bytes.subarray(0, end);
}
```
